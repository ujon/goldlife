# PRD2 - 사이(SAI)

> v0.4 · 2026-05-31
> 목적: AI 코딩 에이전트가 이 문서를 보고 MVP 제품을 만들 수 있을 정도로, 서비스 흐름·온보딩·AI/API 사용·데이터 저장 요구사항을 정리한다.

## 0. 제품 정의

### 0.1 한 줄 정의

사이(SAI)는 사용자가 가진 **시간과 총 예산** 안에서 혼자, 친구, 커플, 가족, 동료/모임이 무엇을 하면 좋을지 추천해주는 AI 기반 활동 추천 서비스다.

### 0.2 핵심 컨셉

사용자는 먼저 로그인하고 기본 성향 온보딩을 완료한다. 이후 추천을 받을 때마다 현재 위치, 함께하는 사람, 사용 가능한 시간, 총 예산을 입력받고, AI는 그 외 맥락(날씨·상황·동행 제약 등)에 대해서만 추가 질문 1-2개를 던진 뒤, 사용 가능한 API를 적극 활용해 실행 가능한 추천 3개를 만든다.

### 0.3 서비스 이름 의미

- **사이**: 시간과 돈 사이에서 가장 괜찮은 선택지를 찾아준다.
- **SAI**: Situation-aware AI. 사용자의 상황을 이해하는 AI 추천.
- 사람 사이: 혼자뿐 아니라 친구, 커플, 가족, 동료/모임 같은 관계 맥락을 반영한다.

### 0.4 MVP 원칙

- 로그인은 필수다.
- 로그인 방식은 이메일 + 비밀번호다.
- 이메일 검증은 MVP에서 하지 않는다.
- 사용자의 기본 성향과 추천 히스토리는 DB에 저장하고 다음 추천에 활용한다.
- 시간과 예산은 온보딩에서 묻지 않는다. 추천 직전에 매번 입력받는다.
- 추천은 항상 3개를 보여준다.
- 추천 카테고리명은 AI가 상황에 맞게 생성한다.
- 예약/구매/상세 링크는 가능하면 제공한다. 링크가 없어도 정보 카드로 추천 가능하다.
- 아기 동반 같은 동행 제약이 있으면 부모/사용자 취향보다 동반자의 안전과 편의를 먼저 본다.
- 디자인은 Duolingo처럼 귀엽고 가볍고 친근한 코치형 UI를 참고한다.

## 1. 화면절차

화면절차는 큰 단계 중심으로 작성한다. 각 화면의 세부 UI 상태는 구현 단계에서 컴포넌트 명세로 분리한다.

```text
1. 로그인/회원가입
2. 위치 권한 요청
3. 고정 온보딩 질문
4. 홈/추천 시작
5. 오늘의 상황 선택
6. 사용 가능 시간 입력
7. 총 예산 입력
8. AI 동적 추가 질문
9. 추천 생성
10. 추천 결과 3개 확인
11. 상세/예약/구매/길찾기 이동
12. 피드백 저장
```

### 1.1 로그인/회원가입

**목적**  
사용자 프로필과 추천 히스토리를 DB에 연결하기 위해 앱 진입 시 인증을 먼저 수행한다.

**요구사항**

- 이메일 + 비밀번호로 회원가입한다.
- 이메일 + 비밀번호로 로그인한다.
- 이메일 검증은 하지 않는다.
- 비밀번호 재설정은 MVP 범위 밖이다.
- 로그인 성공 후 사용자 ID를 기준으로 온보딩 상태와 추천 히스토리를 조회한다.

### 1.2 위치 권한 요청

**목적**  
기본 추천을 현재 위치 기반으로 제공한다.

**요구사항**

- 앱 첫 진입 직후 위치 권한을 요청한다.
- 권한 허용 시 현재 위치를 추천 기준 위치로 저장한다.
- 권한 거부 시 fallback으로 직접 지역 입력 또는 기본 지역을 사용한다.
- 위치는 추천 품질의 핵심 입력값이지만, 과도한 위치 히스토리는 저장하지 않는다.

**카피 톤 예시**

```text
근처에서 재밌는 거 찾아오려면 위치가 필요해!
괜찮아, 싫으면 지역을 직접 골라도 돼.
```

### 1.3 고정 온보딩 질문

**목적**  
사용자의 기본 취향 프로필을 만든다. 시간과 예산은 온보딩에서 묻지 않는다.

**질문 방식**

- 로그인 직후 최초 1회 진행한다.
- 질문은 고정이다.
- 질문 수는 5-6개다.
- 화면 중앙에 AI 캐릭터를 두고, 캐릭터 위 말풍선이 사용자에게 질문하듯이 표시한다.
- 답변은 음성 답변과 키보드 직접 입력으로 받는다. 온보딩 선택지 버튼은 노출하지 않는다.
- 말로 답하기는 온보딩 진입 시 기본으로 켜진다.
- 질문 TTS는 각 온보딩 질문 화면에서 최초 1회만 실행한다.
- 캐릭터를 누르면 질문을 다시 읽지 않고 말로 답하기만 다시 시작할 수 있다.
- 일정 시간 동안 음성 응답이 없으면 말로 답하기는 pause 상태가 된다.
- 별도 스피커 버튼은 제공하지 않는다. 스피커가 켜진 상태에서는 질문을 TTS로 읽고, 아니면 TTS를 건너뛰고 바로 음성 답변을 기다린다.
- 음성 답변 대기 중에는 캐릭터 주변에 pulse/listening 효과를 표시한다.
- Voice 기능은 STT는 브라우저 Web Speech API(`SpeechRecognition`/`webkitSpeechRecognition`)를 사용하고, 온보딩 질문 TTS는 Supertone Ariel voice와 `sona_speech_2_flash` 모델을 서버 프록시로 호출한다. Supertone TTS를 사용할 수 없으면 오류 상태를 표시한다.
- 사용자가 말하면 전사된 답변을 입력칸에 보여주고, 가능한 선택지와 자동 매칭한다.
- 사용자는 언제든 키보드로 직접 답변을 입력하고 반영할 수 있다.
- 선택지로 표현되지 않는 말/문장 답변은 질문 원문과 답변 원문을 함께 DB에 저장하고 추천 알고리즘에서 취향 신호로 사용한다.
- 단, MBTI 질문은 예외다. 음성/텍스트 모두 16개 MBTI 유형 또는 `unknown`으로 매핑되는 "잘 모르겠어"만 입력할 수 있다.
- 톤은 귀여운 코치형 + 친구형이다.
- 아기 동반 같은 동행 제약은 매번 달라지므로 온보딩 고정 질문에 넣지 않고, 추천 직전 상황 선택(1.5)에서 세션 단위로 받는다.

**반드시 파악할 축**

- 활동 성향
- 소비 성향
- 이동 성향
- MBTI

### 1.4 홈/추천 시작

**목적**  
사용자가 지금 쓸 수 있는 시간과 돈으로 추천을 시작한다.

**요구사항**

- 추천 시작 버튼을 제공한다.
- 최근 추천 히스토리가 있으면 간단히 보여줄 수 있다.
- 온보딩 미완료 사용자는 온보딩으로 보낸다.

### 1.5 오늘의 상황 선택

**목적**  
같은 사용자라도 매번 함께하는 사람이 달라질 수 있으므로 추천 직전에 상황을 받는다.

**선택지**

- 혼자
- 친구
- 커플
- 가족
- 동료/모임

**동행 제약 (선택)**

- 상황을 고른 뒤, 아기 동반 같은 동행 제약을 추가로 받을 수 있다. 동행 제약은 상황(혼자/친구/커플/가족/동료·모임)과 직교하며 어떤 상황에도 얹힐 수 있다.
- `아기(영유아) 동반` 여부를 가볍게 묻고, 동반이면 핵심 편의(유모차 이동, 수유실, 기저귀 교체 공간, 주차)를 체크박스로 선택받는다. 이 입력이 동행 제약의 단일 출처이며, AI 동적 질문(1.8)은 이미 받은 값을 보강·세분화만 한다.
- 동행 제약은 추천 필터링의 핵심 입력으로 사용한다. 아기 동반이면 hard filter(필수 필터)로 먼저 적용한다.
- 이 문서에서 '아기 인프라'는 유모차 진입·수유실·기저귀 교체·주차 등 편의시설을 통칭한다.
- MVP의 동행 제약은 아기(영유아) 동반 1종으로 한정한다.

**저장**

- 추천 세션별로 저장한다.
- 사용자 기본 프로필에 고정 저장하지 않는다.
- 아기 동반 같은 동행 제약도 세션 단위로 저장한다.

### 1.6 사용 가능 시간 입력

**목적**  
추천 결과가 단일 활동인지 코스형인지 결정하는 핵심 조건이다.

**입력 방식**

- 기본은 선택 칩이다.
- 직접 입력도 가능하다.

**예시**

- 1시간
- 2-3시간
- 반나절
- 하루
- 이번 주말
- 직접 입력: 오늘 3시부터 7시까지

### 1.7 총 예산 입력

**목적**  
사용자가 이번 활동 전체에 쓸 수 있는 총 금액을 받는다.

**입력 방식**

- 기본은 선택 칩이다.
- 직접 입력도 가능하다.

**예시**

- 1만원
- 3만원
- 5만원
- 10만원
- 15만원+
- 직접 입력

**정책**

- 예산은 총 예산 기준이다.
- 여러 명이어도 총 예산으로 계산한다.
- UI에서는 참고로 1인당 금액을 계산해 보여줄 수 있다.

### 1.8 AI 동적 추가 질문

**목적**  
현재 상황에 따라 추천 품질을 높이기 위한 맥락을 보충한다.

**정책**

- 추천 전 추가 질문은 1-2개만 허용한다.
- 시간과 예산은 이미 고정 질문으로 받았으므로 AI가 다시 묻지 않는다.
- 1.5에서 이미 받은 상황·동행 제약(아기 동반 여부와 핵심 편의)도 다시 묻지 않고, 필요하면 세부 조건만 보강한다.
- 질문은 현재 날씨, 사용자 프로필, 위치, 상황, API 후보 상태에 따라 달라진다.

**예시**

```text
오늘 비가 와. 실내 위주로 볼까, 그래도 밖도 괜찮아?
친구랑이면 수다 위주가 좋아, 같이 뭔가 하는 게 좋아?
예산을 아끼는 게 중요해, 아니면 한 곳에 몰아서 써도 돼?
아기랑 같이 가는구나. 유모차로 이동해야 해, 아니면 아기띠도 괜찮아?
```

### 1.9 추천 결과 3개 확인

**목적**  
사용자가 시간과 돈을 쓰기 전에 비교 가능한 선택지를 제공한다.

**정책**

- 추천은 항상 3개다.
- 3개 추천의 라벨은 AI가 상황에 맞게 생성한다.
- 추천 카드는 이유를 먼저 보여준다.
- 활동 하나도 가능하고, 코스형도 가능하다.
- 시간에 따라 결과 형태가 달라진다.

**시간별 추천 형태**

| 사용 가능 시간 | 추천 형태                         |
| -------------- | --------------------------------- |
| 1-2시간        | 단일 활동 가능                    |
| 2-4시간        | 활동 1개 + 카페/식사 등 미니 코스 |
| 반나절 이상    | 여러 장소를 묶은 코스형           |
| 하루/주말      | 시간표형 코스                     |

> 아기 동반 세션은 같은 시간이라도 이동·체류 부담을 낮춰 더 짧은 동선의 코스로 구성한다.

**추천 카드 우선순위**

1. 왜 이 추천이 맞는지
2. 시간 사용 계획
3. 예상 비용
4. 위치/이동
5. 날씨 적합도
6. 동행 적합도 (아기 동반 시 유모차/수유실/기저귀 교체/주차 등)
7. 상세/예약/구매/길찾기 CTA

### 1.10 피드백 저장

**목적**  
추천 결과에 대한 반응을 다음 추천에 활용한다.

**피드백 방식**

- 좋아요
- 별로예요
- 선택적 이유

**별로예요 이유 예시**

- 비싸요
- 멀어요
- 시간이 안 맞아요
- 취향이 아니에요
- 너무 뻔해요
- 예약이 어려워요
- 아기랑 가기 불편해요
- 아기 편의시설이 부족해요

**좋아요 이유 예시**

- 딱 내 취향
- 예산이 좋아
- 가까워
- 새로워 보여
- 아기랑 가기 편해 보여

## 2. 온보딩절차

온보딩은 추천 조건을 묻는 단계가 아니다. 로그인 후 사용자 기본 성향과 MBTI를 파악해 DB에 저장하는 절차다. 아기 동반 같은 동행 제약은 매번 달라질 수 있으므로 온보딩이 아니라 추천 직전 세션에서 받는다.

### 2.1 온보딩 목표

- 사용자의 활동 성향, 소비 성향, 이동 성향, MBTI를 저장한다.
- 이후 추천 전 동적 질문 수를 줄인다.
- 추천 이유를 개인화할 수 있게 한다.
- Duolingo처럼 부담 없는 질문형 경험을 만든다.

### 2.2 온보딩 질문 구성

| 단계        | 질문 예시                             | 선택지 예시                          | 파악하려는 것                      | 저장 필드 예시         |
| ----------- | ------------------------------------- | ------------------------------------ | ---------------------------------- | ---------------------- |
| 활동 성향 1 | 쉬는 날엔 보통 뭐 하면서 보내?        | 맛집, 전시, 체험, 산책, 집에서 쉬기  | 선호 활동 카테고리                 | `activity_preferences` |
| 활동 성향 2 | 새로운 걸 해보는 편이야?              | 완전 좋아, 가끔 좋아, 익숙한 게 좋아 | 새 경험 선호도                     | `novelty_preference`   |
| 소비 성향 1 | 돈 쓸 때 어떤 쪽이 편해?              | 가성비, 적당히, 프리미엄             | 가격 민감도                        | `spending_style`       |
| 소비 성향 2 | 실패 없는 선택이 좋아, 모험도 괜찮아? | 실패 회피, 균형, 모험 선호           | 리스크 성향                        | `risk_tolerance`       |
| 이동 성향   | 이동은 어느 정도 괜찮아?              | 가까운 곳, 대중교통 가능, 멀어도 됨  | 이동 허용도                        | `mobility_preference`  |
| MBTI        | MBTI 알려줄래?                        | 16개 MBTI 유형, 잘 모르겠어          | 성격 기반 추천 톤과 활동 선호 보정 | `mbti_type`            |

MBTI는 온보딩 필수 항목이다. 사용자가 자신의 유형을 모르면 `unknown`으로 저장하되, 질문 자체를 건너뛰지는 않는다.

### 2.3 온보딩 UI와 입력 방식

온보딩은 AI 캐릭터와 대화하는 느낌을 주되, 사용자가 빠르게 끝낼 수 있어야 한다.

**레이아웃**

- 화면 중앙에 AI 캐릭터를 배치한다.
- 질문은 캐릭터 위 말풍선에 표시한다.
- 말풍선에는 현재 질문과 짧은 리액션을 함께 보여줄 수 있다.
- 온보딩 선택지 버튼은 표시하지 않는다.
- 직접 입력 필드는 캐릭터 아래에 배치한다.
- 별도 입력 반영 버튼은 두지 않는다. 입력칸에 답변이 들어오면 하단 CTA를 활성화하고, CTA를 누를 때 답변 저장과 다음 단계 이동을 함께 처리한다.
- 주요 CTA는 화면 하단 도달권에 둔다.
- 진행 상태는 상단 좌측에 앱 아이콘/이름 없이 선형 progress bar만 표시한다.

**음성 동작**

- 온보딩에는 스피커 토글 버튼을 제공하지 않는다.
- 말로 답하기는 기본으로 켜져 있으며, 온보딩 질문마다 자동으로 음성 답변을 기다린다.
- 스피커가 켜져 있으면 현재 질문을 TTS로 읽어준다. 단, 질문 읽기는 질문 화면당 최초 1회만 실행하고, 캐릭터 재탭 시에는 STT만 다시 시작한다. 스피커가 꺼져 있거나 TTS를 사용할 수 없으면 읽기를 건너뛰고 STT를 시작한다.
- STT는 브라우저 Web Speech API로 한정해 음성 원본을 외부 서버로 보내지 않는다. TTS는 온보딩 질문 텍스트만 Supertone 서버 프록시로 전달하고, 실패 시 오류 상태를 표시한다.
- 음성 응답이 일정 시간 없으면 말로 답하기를 pause하고, 캐릭터를 누르면 다시 시작한다.
- 음성 대기 중 캐릭터는 listening 상태가 되고, 주변 pulse/ring 효과로 대기 중임을 보여준다.
- 사용자가 말하면 전사 텍스트를 직접 입력 필드에 반영한다.
- 전사 텍스트가 선택지와 매칭되면 해당 답변을 선택 처리한다.
- 매칭이 애매하면 전사 텍스트는 입력 필드에 남기고, 사용자가 수정하거나 선택지로 답할 수 있게 한다.
- STT 실패 시 압박하지 않고 직접 입력으로 자연스럽게 폴백한다. TTS 실패 시 Supertone 오류 상태를 표시한다.

**입력 정책**

- 키보드 직접 입력은 항상 가능하다.
- MBTI 외 질문의 직접 입력값은 가능한 경우 정해진 저장 필드 값으로도 정규화한다.
- MBTI 외 질문의 직접 입력값이 선택지와 매칭되지 않아도 질문 원문, 답변 원문, 입력 출처, 입력 시각을 저장한다.
- 저장된 자유 문장 답변은 추천 후보 검색, 랭킹, 추천 이유 생성에 사용한다.
- MBTI 질문의 직접 입력값은 `INTP`처럼 완성된 4글자 유형 또는 "잘 모르겠어"만 허용한다. 문장형 답변이나 미완성 유형은 하단 CTA를 활성화하지 않고 저장하지 않는다.
- 음성 원본은 저장하지 않는다.
- MBTI 외 전사 텍스트는 사용자가 하단 CTA를 누르면 자유 문장 답변으로 저장할 수 있다.
- MBTI 전사 텍스트는 허용된 유형/unknown으로 정확히 매칭될 때만 저장한다.

### 2.4 온보딩 톤

AI 캐릭터는 귀여운 코치형 + 친구형이다.

**말투 원칙**

- 짧게 묻는다.
- 반말을 기본으로 하되 무례하지 않다.
- 실패해도 압박하지 않는다.
- 사용자의 답변에 가볍게 리액션한다.

**예시**

```text
좋아, 너 취향부터 살짝 알아볼게.
새로운 곳 도전하는 편이야, 익숙한 곳이 편해?
오케이. 그럼 너무 실험적인 추천은 살짝 줄여둘게.
```

## 3. 서비스

### 3.1 사용자 문제

- 시간이 생겨도 무엇을 할지 결정하기 어렵다.
- 예산 안에서 할 수 있는 선택지를 찾는 데 시간이 오래 걸린다.
- 맛집, 액티비티, 날씨, 지도, 예약 정보를 여러 서비스에서 따로 봐야 한다.
- 추천 리스트는 많지만 왜 나에게 맞는지 설명이 부족하다.
- 함께하는 사람이 달라질 때마다 적절한 추천 기준도 달라진다.
- 아기 동반처럼 동행 제약이 있으면 안전·편의까지 따져야 해서 적합한 장소 찾기가 훨씬 까다롭다.

### 3.2 사용자 가치

- 시간과 돈 안에서 실행 가능한 선택지를 빠르게 받는다.
- 현재 위치와 날씨를 반영한 현실적인 추천을 받는다.
- 사용자의 성향과 히스토리를 반영해 점점 나아지는 추천을 받는다.
- 추천 이유, 시간 계획, 비용, 이동 정보를 한 카드에서 확인한다.
- 예약/구매/상세 링크가 있으면 바로 행동으로 이어갈 수 있다.
- 아기 동반처럼 제약이 큰 상황에서도 안전·편의를 먼저 거른 현실적인 추천을 받는다.

### 3.3 서비스 대상

사이는 커플 전용 서비스가 아니다. 다음 상황을 모두 지원한다.

- 혼자
- 친구
- 커플
- 가족
- 동료/모임

각 상황에는 아기 동반 같은 동행 제약이 더해질 수 있고, 사이는 이를 추천에 핵심 조건으로 반영한다.

### 3.4 추천 범위

**기본은 오프라인 중심**

- 맛집
- 카페
- 전시
- 공연
- 액티비티
- 여행
- 원데이클래스
- 쇼핑
- 체험

**fallback으로 집/온라인 추천 가능**

- 집에서 할 수 있는 활동
- OTT/게임
- 배달
- 온라인 클래스
- 온라인 쇼핑

**fallback 조건**

- 시간이 너무 짧다.
- 예산이 너무 낮다.
- 날씨가 좋지 않다.
- 사용자가 이동을 싫어한다.
- 근처 API 결과가 부족하다.
- 아기 동반 등 동행 제약을 만족하는 적합 후보가 부족하다.

### 3.5 서비스 모듈

| 모듈                    | 역할                                                                                                     |
| ----------------------- | -------------------------------------------------------------------------------------------------------- |
| Auth                    | 이메일/비밀번호 회원가입과 로그인                                                                        |
| Location                | 현재 위치 권한 요청, 현재 위치 저장, 지역 fallback                                                       |
| Onboarding              | 고정 질문으로 기본 성향 프로필 생성                                                                      |
| Recommendation Intake   | 매 추천마다 상황, 시간, 총 예산 입력                                                                     |
| AI Follow-up            | AI가 현재 상황 기반 추가 질문 1-2개 생성. 1.5에서 받은 아기 동반 등 동행 제약의 세부 조건 보강 질문 포함 |
| API Candidate Collector | 외부 API에서 활동/맛집/날씨/지도/트렌드 후보 수집. 아기 동반 시 편의시설(유모차/수유실/주차) 정보 포함   |
| AI Orchestrator         | API 호출 순서와 추천 전략 결정. 동행 제약을 hard filter로 우선 적용                                      |
| Recommendation Composer | 추천 3개와 카드 JSON 생성                                                                                |
| Outbound Link           | 상세/예약/구매/길찾기 링크 제공                                                                          |
| Feedback                | 좋아요/별로예요와 이유 저장                                                                              |

### 3.6 차별점

- 단순 검색이 아니라 시간, 돈, 위치, 날씨, 성향, 동행 상황을 함께 본다.
- 여러 API를 추천 파이프라인에 적극적으로 연결한다.
- AI가 추천 라벨과 추천 이유를 상황에 맞게 생성한다.
- DB에 저장한 성향과 히스토리를 다음 추천에 반영한다.
- 아기 동반 같은 동행 제약을 hard filter로 적용해, 취향보다 안전·편의를 먼저 본다.

## 4. 유스케이스

### 4.1 UC-001 - 친구와 반나절 놀기

**사용자 상황**

- 친구와 반나절 시간이 있다.
- 총 예산은 10만원이다.
- 현재 위치 기반으로 가까운 곳을 찾고 싶다.
- 날씨와 이동 거리를 고려하고 싶다.

**추천 목표**

- 친구 둘이 함께 이야기하거나 같이 체험할 수 있는 활동을 추천한다.
- 예산 안에서 액티비티와 식사/카페를 조합한다.
- 날씨가 나쁘면 실내 활동을 우선한다.

**AI 동적 질문 예시**

```text
친구랑이면 수다 위주가 좋아, 같이 뭔가 하는 활동이 좋아?
```

**추천 예시**

- 실내 원데이 클래스 + 캐주얼 다이닝
- 전시 + 디저트 카페
- 방탈출 + 간단한 저녁

### 4.2 UC-002 - 아기가 있는 등산 좋아하는 신혼부부

**사용자 상황**

- 사용자는 신혼부부다.
- 아기가 함께 있다.
- 부부는 등산이나 자연 속 활동을 좋아한다.
- 하지만 아기 때문에 난이도, 이동 시간, 날씨, 편의시설이 중요하다.
- 사용 가능한 시간과 총 예산 안에서 좋은 장소를 찾고 싶다.

**핵심 문제**

등산을 좋아하지만 일반 등산 코스는 아기 동반에 위험하거나 피로도가 높을 수 있다. 따라서 AI는 "등산 취향"을 그대로 적용하지 않고, 아기 동반 제약을 우선 고려해 자연 친화적인 낮은 난이도 코스로 변환해야 한다.

**아기 동반 우선순위 규칙**

아기가 포함된 추천에서는 부모의 취향보다 아기 인프라와 안전성을 먼저 본다. 사용자가 등산, 야외, 맛집, 체험을 선호하더라도 아기 동반 인프라가 부족하면 추천에서 제외하거나 낮게 랭킹한다.

우선순위는 다음과 같다.

1. 아기 안전과 필수 인프라: 유모차/아기띠 적합성, 수유실, 기저귀 교체 공간, 화장실, 실내 대피 공간
2. 날씨와 이동 부담: 비/폭염/한파/미세먼지, 이동 시간, 도보 거리, 주차/대중교통 접근성
3. 시간과 총 예산 제약
4. 부모의 취향: 등산, 자연, 맛집, 체험 등

**추천 목표**

- 아기 인프라를 먼저 만족한 후보 안에서 등산 감성을 살린 장소를 추천한다.
- 급경사, 긴 코스, 화장실/수유/기저귀 교체가 어려운 장소는 부모 취향과 잘 맞아도 제외하거나 낮게 랭킹한다.
- 유모차 가능 여부, 아기띠 필요 여부, 주차, 화장실, 실내 대피 가능 공간을 함께 고려한다.
- 날씨가 나쁘면 숲길 대신 실내 식물원, 대형 키즈 프렌들리 카페, 자연사/과학관 등으로 대체한다.

**추천 전 고정 입력 예시**

```json
{
  "situation": "family",
  "availableTime": "half_day",
  "budgetTotal": 120000,
  "location": "current"
}
```

**AI 동적 질문 예시**

```text
아기랑 같이 가는구나. 유모차로 이동해야 해, 아니면 아기띠도 괜찮아?
```

```text
등산 느낌은 살리되 낮은 산책로 위주로 볼게. 수유실이나 기저귀 갈이 공간도 꼭 필요해?
```

**이번 세션의 동행 제약 (1.5 상황 선택에서 받고 AI가 보강)**

```json
{
  "companionConstraints": {
    "hasBaby": true,
    "strollerRequired": true,
    "babyCarrierOk": false,
    "needsNursingRoom": true,
    "needsDiaperChangingRoom": true,
    "preferParking": true
  },
  "activityIntent": {
    "likesHiking": true,
    "convertedPreference": "baby_friendly_nature_walk",
    "avoid": ["steep_trail", "long_hike", "remote_area"]
  }
}
```

**API 활용**

| API/데이터        | 활용                                                                                     |
| ----------------- | ---------------------------------------------------------------------------------------- |
| 날씨 API          | 비, 폭염, 한파, 미세먼지면 야외 산책/등산 후보를 낮추고 실내 fallback 추천               |
| 지도/동선 API     | 현재 위치에서 이동 시간, 주차 가능성, 도보 거리 확인                                     |
| 맛집/카페 API     | 유아 동반 가능, 넓은 좌석, 유모차 진입 가능 후보 우선                                    |
| 액티비티/장소 API | 숲길, 공원, 식물원, 자연 체험, 낮은 난이도 산책 후보 수집                                |
| AI                | 아기 인프라를 hard filter로 적용한 뒤, 등산 선호를 아기 동반 가능한 자연형 코스로 재해석 |

**추천 결과 예시**

1. `아기랑 숲냄새 픽`
   - 유모차 가능한 완만한 숲길
   - 근처 넓은 브런치 카페
   - 이유: 등산 취향은 살리면서 아기 이동 부담을 줄임

2. `아기띠 가능 가벼운 산책픽`
   - 짧은 둘레길
   - 주차장 가까운 식당
   - 이유: 자연 속 활동을 하되 반나절 안에 무리 없이 종료

3. `비 오면 자연 실내픽`
   - 실내 식물원 또는 자연사/과학관
   - 키즈 프렌들리 카페
   - 이유: 날씨가 나쁠 때도 자연 감성을 유지

**추천 카드에 반드시 보여줄 정보**

- 아기 동반 적합 이유
- 이동 시간
- 유모차/아기띠 적합성
- 화장실/수유실/기저귀 교체 가능 여부
- 날씨 적합도
- 예상 비용
- 주차 또는 대중교통 접근성

**피드백 예시**

- 좋아요: `아기랑 가기 편해 보여`
- 좋아요: `등산 느낌이 살아있어`
- 별로예요: `유모차가 힘들 것 같아`
- 별로예요: `아기 편의시설이 부족해`

### 4.3 UC-003 - 혼자 2시간 기분전환

**사용자 상황**

- 혼자 2시간 정도 시간이 있다.
- 총 예산은 3만원이다.
- 멀리 이동하고 싶지 않다.

**추천 목표**

- 현재 위치 근처에서 짧고 부담 없는 활동을 추천한다.
- 단일 활동 추천도 허용한다.
- 카페, 짧은 전시, 산책, 서점, 영화 등을 후보로 둔다.

**AI 동적 질문 예시**

```text
혼자 2시간이면 가볍게 기분전환이 좋겠다. 조용히 쉬고 싶어, 아니면 뭔가 보고 싶어?
```

### 4.4 UC-004 - 아기 동반 가족의 비 오는 날 실내 나들이

**사용자 상황**

- 가족이 함께 있고 아기가 동반한다.
- 사용 가능한 시간은 2-3시간이다.
- 총 예산은 6만원이다.
- 오늘은 비가 오고 미세먼지도 있다.

**핵심 문제**

짧은 시간, 나쁜 날씨, 아기 동반이 한꺼번에 겹친다. 야외·이동이 많은 후보는 제외하고, 실내이면서 아기 편의시설이 갖춰진 가까운 장소를 우선해야 한다. UC-002가 야외(등산) 취향을 아기 동반용으로 변환하는 사례라면, 이 케이스는 짧은 시간·악천후에서의 실내 동선 구성을 다룬다.

**아기 동반 우선순위 적용**

1. 실내 + 아기 편의시설(수유실/기저귀 교체/유모차 진입)
2. 가까운 이동 거리와 주차 가능성
3. 시간과 총 예산 제약
4. 가족 취향

**AI 동적 질문 예시**

```text
비도 오고 아기랑이면 실내가 편하겠다. 수유실이나 기저귀 갈이 공간은 꼭 필요해?
```

**추천 예시**

1. `비 와도 OK 실내픽` - 유모차 진입 가능한 키즈 프렌들리 카페 + 실내 놀이 공간
2. `가까운 실내 체험픽` - 차로 10분 내 영유아 동반 가능한 실내 전시/체험관
3. `짧고 편한 가성비픽` - 수유실·기저귀 교체가 되는 몰 안에서 식사 + 휴식

**추천 카드에 반드시 보여줄 정보**

- 실내 여부와 날씨 적합도
- 아기 편의시설(수유실/기저귀 교체/유모차)
- 이동 시간과 주차
- 예상 비용

## 5. API 활용 전략

### 5.1 추천 파이프라인

```text
1. 로그인 사용자 프로필 조회
2. 현재 위치 확인
3. 날씨 API 조회
4. 추천 전 입력값 수집: 상황, 시간, 총 예산
5. AI 동적 질문 1-2개 생성
6. 사용자 답변으로 이번 세션 선호 업데이트
7. 트렌드/API 후보 수집
8. 시간, 예산, 거리, 날씨, 예약 가능성으로 후보 필터링
   - 단, 아기 동반 세션은 아기 인프라와 안전성을 필수 필터로 먼저 적용
9. 사용자 성향과 피드백 히스토리로 후보 랭킹
   - 단, 아기 동반 세션은 아기 인프라 점수가 사용자 취향 점수보다 우선
10. 추천 3개 생성
    - 단, 아기 동반 세션은 카드에 동행 편의(유모차/수유실/기저귀 교체/주차) 정보를 함께 채운다
11. 상세/예약/구매/길찾기 링크 연결
12. 피드백 저장
```

| ----------------------- | ------------------------------------------------------------------------------------------------------- |
| OpenAI                  | 추천 오케스트레이션, API 호출 순서 판단, 최종 추천 카드 JSON 생성                                       |
| EXAONE                  | 한국어 대화형 온보딩, 성향 파악, 귀여운 코치/친구 톤 응답 생성                                          |

### 5.3 날씨 API 사용

날씨는 추천 품질에 직접 반영한다.

**예시 정책**

- 비/눈: 실내 활동, 예약 가능한 맛집, 가까운 동선 가중치 상승
- 폭염/한파: 실내, 이동 짧은 코스 가중치 상승
- 미세먼지 나쁨: 실내 활동 가중치 상승
- 날씨 좋음: 산책, 야외, 근교 활동 후보 포함
- 아기 동반: 비/폭염/한파/미세먼지 가중치를 더 강하게 적용하고 실내·짧은 동선을 우선

### 5.4 tobl 페르소나 토론 사용 정책

페르소나 토론은 필수 화면이 아니라 선택적 연출이다.

- 첫 추천 또는 데모에서는 보여줄 수 있다.
- 일반 흐름에서는 접거나 빠르게 넘길 수 있다.
- 내부적으로는 후보 비교와 추천 이유 생성에 활용 가능하다.
- UI 표현은 `AI 친구들이 잠깐 회의했어` 정도로 가볍게 한다.

## 6. 데이터 저장 요구사항

### 6.1 저장할 사용자 정보

사용자 정보는 DB에 저장하고 추천에 활용한다.

**저장 대상**

- 이메일
- 비밀번호 해시
- 온보딩 완료 여부
- 활동 성향
- 소비 성향
- 이동 성향
- MBTI
- 온보딩 자유 문장 답변. 질문 원문, 답변 원문, 입력 출처(voice/text), 입력 시각
- 최근 위치 또는 자주 쓰는 위치
- 추천 세션 히스토리
- 추천 카드 히스토리
- 클릭한 카드
- 좋아요/별로예요 피드백
- 피드백 이유
- 추천 세션별 동행 제약. 예: 아기 동반, 유모차 필요, 주차 선호

- 음성 원본
- 사용자가 하단 CTA로 확정하지 않은 임시 음성 전사 텍스트. 단, 사용자가 CTA로 확정한 MBTI 외 답변 텍스트는 온보딩 자유 문장 답변으로 저장한다.
- 결제정보
- 불필요한 상세 위치 로그
- 이메일 인증 상태. MVP에서는 검증하지 않음

## 8. 기능 요구사항

### 8.1 P0

- 사용자는 이메일/비밀번호로 회원가입할 수 있다.
- 사용자는 이메일/비밀번호로 로그인할 수 있다.
- 로그인 후 위치 권한을 요청한다.
- 사용자는 고정 온보딩 질문을 완료할 수 있다.
- 사용자는 온보딩에서 MBTI를 입력할 수 있다.
- 온보딩 질문은 중앙 캐릭터와 캐릭터 위 말풍선으로 표시된다.
- 사용자는 온보딩 질문에 음성 또는 키보드 직접 입력으로 답할 수 있다. 온보딩 선택지 버튼은 노출하지 않는다.
- 말로 답하기는 온보딩 진입 시 기본으로 켜지고, 캐릭터를 누르면 다시 시작할 수 있다.
- 온보딩 질문 TTS는 질문 화면당 최초 1회만 실행되며, 캐릭터 재탭은 음성 답변 대기만 재개한다.
- 일정 시간 동안 음성 응답이 없으면 말로 답하기는 pause 상태가 된다.
- 스피커 버튼은 노출하지 않는다. 스피커가 켜진 상태에서는 질문을 음성으로 읽고, 아니면 읽기를 건너뛴다.
- 음성 답변 대기 중에는 캐릭터 주변에 pulse/listening 효과를 표시한다.
- 음성 답변은 전사 후 입력칸에 표시되고 가능한 선택지와 자동 매칭된다.
- MBTI 외 온보딩 자유 문장 답변은 질문 원문과 답변 원문 그대로 DB에 저장된다.
- 저장된 온보딩 자유 문장 답변은 추천 후보 검색, 랭킹, 추천 이유 생성에 반영된다.
- MBTI는 음성/텍스트 모두 16개 유형 또는 "잘 모르겠어"만 저장할 수 있다.
- 온보딩 결과는 DB에 저장된다.
- 사용자는 추천 시작 시 상황을 선택할 수 있다.
- 사용자는 상황 선택 시 아기 동반 여부 등 동행 제약을 입력할 수 있다.
- 사용자는 사용 가능한 시간과 총 예산을 입력할 수 있다.
- AI는 추천 전 동적 질문을 1-2개 생성한다.
- 시스템은 날씨 API를 호출해 추천에 반영한다.
- 시스템은 외부 API 후보를 수집해 추천 3개를 생성한다.
- 추천 카드는 이유, 시간, 비용, 위치/이동, 날씨 적합도를 포함한다.
- 사용자는 추천 카드에 좋아요/별로예요 피드백을 남길 수 있다.
- 피드백은 DB에 저장된다.
- 가족/아기 동반 같은 동행 제약은 추천 세션에 저장하고 추천 필터링에 반영한다.
- 아기가 포함된 세션에서는 사용자 취향보다 아기 인프라와 안전성 조건을 우선 적용한다.

### 8.2 P1

- 상세/예약/구매/길찾기 링크를 가능한 경우 제공한다.
- 사용자는 별로예요 이유를 선택할 수 있다.
- 추천 라벨은 AI가 상황에 맞게 생성한다.
- 이전 추천 히스토리를 다음 추천에 반영한다.
- 외부 API 실패 시 fallback 추천을 제공한다.
- 아기 동반 추천 카드에는 유모차/수유실/기저귀 교체/주차 같은 편의 조건을 표시한다.
- 아기 인프라 정보가 부족한 후보는 `정보 부족`으로 표시하거나 추천 순위를 낮춘다.

### 8.3 MVP 제외

- 이메일 검증
- 비밀번호 재설정
- 소셜 로그인
- 자체 결제
- 예약 확정 조회
- 음성 원본 저장
- 장기 위치 로그 저장

## 9. 디자인 방향

### 9.1 참고 방향

Duolingo처럼 귀엽고, 가볍고, 질문에 답하고 싶게 만드는 디자인을 참고한다.

### 9.2 UI 톤

- 둥근 형태
- 밝고 친근한 컬러
- 캐릭터 또는 말풍선 중심 온보딩
- 온보딩 질문은 캐릭터 위 말풍선으로 보여주고, 캐릭터는 화면 중앙에 둔다.
- 음성 대기 중에는 캐릭터 주변 pulse/ring 효과로 listening 상태를 명확히 보여준다.
- 진행 상태는 상단 좌측에 앱 아이콘/이름 없이 선형 progress bar만 표시한다.
- 짧은 문장
- 즉각적인 리액션
- 진행감이 보이는 단계 UI
- 동행 편의 정보(예: 아기 동반 시 유모차/수유실/주차)는 카드에서 배지로 한눈에

### 9.3 피해야 할 것

- 광고 리스트처럼 보이는 추천 화면
- 정보가 너무 빽빽한 대시보드
- 사용자를 평가하는 듯한 성향 질문
- 권한 요청을 딱딱하게 보여주는 흐름

## 11. 구현 우선순위

AI 코딩 에이전트는 아래 순서로 MVP를 만들면 된다.

1. 이메일/비밀번호 로그인과 회원가입
2. 사용자 프로필 DB 모델
3. 위치 권한 요청과 fallback 지역 입력
4. MBTI를 포함한 고정 온보딩 질문 UI와 저장
5. 추천 시작 플로우: 상황(아기 동반 등 동행 제약 포함), 시간, 총 예산 입력
6. AI 동적 질문 mock
7. 날씨 API 또는 mock 날씨 데이터 연결
8. 추천 3개 mock 생성
9. 외부 API 후보 수집 래퍼 추가
10. AI 추천 카드 JSON 생성
11. 피드백 저장
12. 이전 피드백을 다음 추천에 반영
